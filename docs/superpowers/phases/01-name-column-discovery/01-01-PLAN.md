# Phase 1, Plan 1: Name Column Discovery

## Objective

Auto-discover every column in the DuckLake that contains person or entity names. Produce a `NAME_SOURCES` registry — a Python data structure mapping each table to its name columns, column patterns, and extraction strategy. This replaces the hardcoded 15-table list in `entity_resolution.py` with a comprehensive ~50-table auto-discovered registry.

## Scope

- **Estimate**: Small (1-2 hours)
- **Files modified**: 1 new file, 1 modified
- **Risk**: Low — read-only introspection, no data mutation

## Context

### Prior Art
- `entity_resolution.py` has a hardcoded `NAME_SOURCES` list with 15 tables
- We ran a discovery query that found ~50 tables with name-like columns
- Column patterns vary: `first_name`/`last_name` (structured), `name` (combined "LASTNAME, FIRSTNAME"), `ownername` (PLUTO), `respondent_name` (free text), `license_holder_name` (combined)
- Some tables have address columns alongside names (useful for Splink matching)

### Key Decisions
- Include ALL person-name columns (50+ tables), not just the obvious ones
- Categorize each table's column pattern (structured, combined, free-text)
- Exclude non-person names (park_name, station_name, agency_name, etc.)
- Exclude `_staging` schemas and `_dlt_*` system tables
- Exclude `v_*` view duplicates

## Tasks

### Task 1: Run discovery query on live lake
Connect to DuckLake via MCP and run the comprehensive column discovery query. Capture all tables with person-name columns, grouped by extraction pattern.

**Checkpoint**: Print table count per pattern type. Expected: ~50 tables total.

### Task 2: Classify column patterns
For each discovered table, classify into one of these extraction patterns:
- **STRUCTURED**: Has separate `first_name` and `last_name` columns
- **COMBINED_COMMA**: Single `name` column with "LASTNAME, FIRSTNAME" format (ACRIS, campaign finance)
- **COMBINED_SPACE**: Single name column with "FIRSTNAME LASTNAME" format
- **OWNER**: Property owner field (may contain LLC names — need filtering)
- **RESPONDENT**: Free-text respondent field (litigation, OATH)
- **MULTI**: Table has multiple name-bearing column sets (e.g., dob_permit_issuance has owner + permittee + site_safety_mgr)

Also tag each table with available address columns (address, city, zip) for Splink matching.

**Checkpoint**: Each table has a pattern classification and address column list.

### Task 3: Build the registry module
Create `src/dagster_pipeline/sources/name_registry.py` containing:
- `NameSource` dataclass with fields: `schema`, `table`, `pattern`, `name_columns`, `address_columns`
- `NAME_REGISTRY: list[NameSource]` — the complete list of all discovered tables
- `get_extraction_sql(source: NameSource) -> str` — generates the SELECT statement for each pattern
- Filter logic to exclude LLC/CORP/INC/TRUST/BANK/CITY OF from owner-pattern tables

**Checkpoint**: `len(NAME_REGISTRY)` matches discovery count. Each entry generates valid SQL.

### Task 4: Validate registry against live data
For each entry in the registry, run `SELECT ... LIMIT 5` against the lake to verify:
- Table exists and is accessible
- Column names are correct
- Extraction SQL produces sensible (last_name, first_name) output
- No crashes on NULL or empty values

**Checkpoint**: All entries validate successfully. Print any failures.

### Task 5: Update entity_resolution.py
Replace the hardcoded `NAME_SOURCES` list in `entity_resolution.py` with an import from `name_registry.py`. Update `_build_name_index()` to use the registry's `get_extraction_sql()` method instead of the inline SQL generation.

**Checkpoint**: `python -c "import ast; ast.parse(...)"` passes. Existing proof-of-concept still works.
