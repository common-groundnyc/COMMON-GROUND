**Bug 1 — S3 credentials**:
DuckLake stores `data_path = s3://ducklake/data/` in Postgres catalog metadata. When DuckDB reads parquet files, it resolves `s3://ducklake/` as `ducklake.s3.amazonaws.com` instead of MinIO at `minio:9000`. Both `CREATE SECRET` with SCOPE and `SET s3_*` global settings have been tried — neither works for DuckLake's internal file reads.

The Dockerfile at `/opt/common-ground/duckdb-server/Dockerfile` installs `duckdb==1.5.0` and `fastmcp[http]==3.1.1`. The `start.sh` launches the Python server. The docker-compose mounts `lakevol:/data/common-ground` and exposes ports 4213/9999.

Root cause hypothesis: DuckLake 0.4 uses its own S3 client that respects `CREATE SECRET` but requires the secret to be created BEFORE the `ATTACH` and may need `SCOPE 's3://'` (global) not `SCOPE 's3://ducklake/'` (bucket-specific). Alternatively, the DuckLake ATTACH statement may accept storage credential options directly.

**Bug 2 — duckpgq not loading**:
The extension file exists at `/root/.duckdb/extensions/v1.5.0/linux_amd64/duckpgq.duckdb_extension`. The LOAD command silently fails — no error logged, but `CREATE PROPERTY GRAPH` syntax is not recognized. The Dockerfile was rebuilt with `--no-cache` which may have pulled a newer DuckDB 1.5.0 build that's incompatible with the cached duckpgq extension binary.

Current extension loading code (line ~516):
```python
for ext in COMMUNITY_EXTS:
    try:
        conn.execute(f"LOAD {ext}")
    except Exception:
        try:
            conn.execute(f"INSTALL {ext} FROM community")
            conn.execute(f"LOAD {ext}")
        except Exception as e:
            print(f"Warning: community extension {ext} unavailable: {e}")
```

Fix approach: Force reinstall with `FORCE INSTALL duckpgq FROM community` to ensure binary matches the running DuckDB version.
